# AGENTS.md

## 项目背景

本项目是「酒水分销商城一期」，包含：
- 微信小程序用户端，技术栈：uni-app。
- PC 管理后台，技术栈请根据仓库现状识别。
- 后端服务，技术栈：Spring Boot。
- 数据库优先 MySQL。
- 接口风格优先 RESTful API。

一期目标是完成可上线运营的商城和传统分销 / 招商奖励闭环，同时严格控制推荐层级展示，避免多层级分销合规风险。

## 核心业务范围

必须支持：

1. 微信登录、注册、强制推荐绑定。
2. 后台创建种子账号和邀请码。
3. 首页轮播、公告、客服入口、四个专区：
   - 主产品专区
   - 招商专区
   - 零售专区
   - 礼包专区
4. 商品、购物车、结算、订单、售后、地址。
5. 实名认证提交与后台人工审核。
6. 余额、DF、酒豆、积分、期权等资产账户和明细。
7. 推客升级、联营商升级、招商奖励、扶商奖励、广告奖励。
8. 提现申请、提现审核、线下打款记录。
9. 内容中心，包括视频、图文、权限、付费内容。
10. PC 管理后台，包括用户、商品、订单、售后、资产、提现、实名、内容、公告、配置、权限、日志。
11. 旧系统数据迁移预留能力。

## 必须遵守的业务约束

### 推荐绑定

除后台创建的初始种子账号外，所有普通新用户注册必须绑定上级。

绑定来源包括：
- 分享小程序入口。
- 分享商品链接。
- 手动输入邀请码。
- 扫描邀请码图片。

如果用户自行搜索小程序进入且无上级信息，不允许完成注册，应提示输入邀请码或通过推荐链接进入。

### 合规展示限制

小程序端只能展示：
- 我的邀请人。
- 直推人员列表。
- 间推人员列表。
- 团队总人数统计。

小程序端禁止展示：
- 三级及以上人员明细。
- 多层级树状图。
- 可逐层展开到三级以上的组织关系。
- “无限层级收益”“团队裂变收益”等营销化文案。

后台可以保存完整推荐链路用于奖励计算、统计、审计，但后台深层查询必须受权限控制，并记录操作日志。

### 资产独立

余额、DF、酒豆、积分、期权必须独立记账。

除 PRD 明确允许的转换外：
- 不允许组合支付。
- 不允许余额 + 酒豆。
- 不允许积分 + 现金。
- 不允许积分 + 酒豆。
- 不允许资产自动互通。

### 奖励和规则配置化

以下规则不得硬编码，必须通过数据库配置表、系统配置或枚举配置集中管理：
- 等级名称。
- 等级顺序。
- 晋升条件。
- 主产品折扣。
- 佣金比例。
- 招商奖励封顶金额。
- 扶商奖励比例。
- 广告奖比例。
- 提现手续费。
- 最小提现金额。
- 售后冻结期。
- 内容观看权限。
- 商品可用支付方式。
- 招商礼包金额。
- 联营商复购佣金比例。

对于 PRD 中标记为“待确认”的规则，不要写死默认值。可以实现配置能力，并在种子数据或配置说明中标注“待业务确认”。

## 一期不做

不要主动实现以下能力，除非用户明确要求：
- 合伙人奖励制度。
- 后台白名单。
- 第三方物流 API。
- 微信订单中心打通。
- 第三方实名认证接口。
- 完整线上退换货物流闭环。
- 评价体系。
- 积分签到、抽奖等玩法。
- 多层级团队树状图。
- 小程序展示三级及以上推荐人员。

## 开发工作方式

执行任何任务前，请先：

1. 阅读仓库结构。
2. 识别前端、后端、管理后台所在目录。
3. 阅读 README、package.json、pom.xml、build.gradle、配置文件、路由文件、数据库迁移文件。
4. 总结当前项目已有能力。
5. 给出分步实施计划。
6. 再开始修改代码。

修改代码时请遵守：

1. 优先复用现有项目结构、命名风格、工具类、响应格式、异常处理方式。
2. 不要大规模重构无关代码。
3. 不要删除已有业务逻辑，除非确认它与 PRD 冲突。
4. 每次任务只完成当前 Prompt 指定范围。
5. 涉及资产、支付、奖励、提现、等级变更时，必须保证事务一致性、幂等性和审计日志。
6. 涉及金额、资产数量，请使用 BigDecimal 或项目中已有的精确数值类型，不要使用 float / double。
7. 所有关键状态必须使用枚举或集中常量，不要散落字符串。
8. 所有后台关键操作必须写操作日志。
9. 所有接口必须做权限校验和参数校验。
10. 代码完成后必须运行相关测试、类型检查或构建命令；如果无法运行，请说明原因。
