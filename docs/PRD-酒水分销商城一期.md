## 1. 文档目标

### 1.1 读者

本文档面向产品经理、UI 设计师、前端工程师、后端工程师、测试工程师和项目负责人。

### 1.2 读完后应能完成的动作

读者应能基于本文档拆分一期开发任务、设计页面、定义接口和数据表、编写测试用例，并明确哪些需求属于一期、二期或待确认范围。

### 1.3 决策优先级

三份材料存在部分差异。PRD 按以下优先级整理：

1. 以 2026-05-12 会议形成的最新业务决策为准。
2. 功能报价清单作为功能覆盖面和页面颗粒度参考。
3. 对仍未最终确认的比例、命名和范围，统一放入“待确认事项”，不得在代码中硬编码。

## 2. 项目背景与目标

平台面向酒水零售、主产品复购和招商礼包销售，提供微信小程序商城、会员等级、推荐绑定、奖励结算、资产账户、内容付费和管理后台能力。

一期目标是完成可上线运营的商城和传统分销/招商奖励闭环，同时严格控制推荐层级展示，规避多层级分销合规风险。

### 2.1 一期核心目标

1. 用户可通过微信小程序浏览、注册、绑定推荐人、购买商品和查看订单。
2. 平台可配置首页、商品专区、商品、内容、公告和基础系统参数。
3. 用户可通过主产品区消费升级为推客，并按规则获得主产品推荐奖励。
4. 用户可通过招商区购买礼包或业绩达标升级为联营商，并触发招商奖励、扶商奖励、广告奖励。
5. 平台支持余额、DF、酒豆、积分等资产的独立管理和明细记录。
6. 管理后台支持实名认证审核、订单处理、售后审核、物流单号录入、提现审核和奖励结算辅助。
7. 小程序端只展示合规允许范围内的推荐关系。

### 2.2 一期不做

1. 合伙人奖励制度不在一期开发，包含生态合伙人、事业合伙人、城市合伙人、副总裁、高级总裁、联合创始人、创始人等。
2. 后台白名单功能暂缓，除非业务重新确认。
3. 不对接政府或第三方实名认证接口，一期采用人工审核。
4. 不对接第三方物流 API，一期由后台人工录入物流单号。
5. 不打通微信小程序订单中心，一期平台订单与微信订单中心独立。
6. 不建设完整线上退换货物流闭环，一期采用线上申请、后台审核、线下处理。
7. 不支持积分加现金、余额加酒豆等组合支付。
8. 不在小程序端展示三级及以上推荐人员明细、树状团队图或多层级关系链。

## 3. 业务范围

### 3.1 端口

| 端口 | 使用对象 | 一期范围 |

| 微信小程序用户端 | 游客、会员、推客、联营商 | 浏览、登录注册、购物、内容、资产、团队、订单、售后、地址、实名认证 |

| 等级 | 定义 | 主要权限 |
| --- | --- | --- |

## 4. 核心原则

### 4.1 强制推荐绑定

除后台创建的初始种子账号外，所有新用户注册必须绑定上级。绑定方式包括：

1. 通过他人分享的小程序入口进入。
2. 通过他人分享的商品链接进入。
3. 手动输入邀请码。
4. 扫描邀请码图片。

若用户自行搜索小程序进入且无上级信息，不能完成注册，应提示用户输入邀请码或通过推荐链接进入。

### 4.2 合规展示限制

小程序端仅展示：

1. 我的邀请人。
2. 直推人员列表。
3. 间推人员列表。
4. 团队总人数统计。

小程序端禁止展示：

1. 三级及以上人员明细。
2. 多层级树状图。
3. 可逐层展开到三级以上的组织关系。
4. 容易表达“无限层级收益”的文案或页面。

后台可以保存完整推荐链路用于奖励计算和风控统计，但后台页面也应避免以树状图或营销化方式展示过深层级。运营查询应以审计、统计、结算为目的，并保留操作日志。

### 4.3 资产独立

余额、DF、酒豆、积分、期权各自独立记账。除明确允许的转换外，不做组合支付，不做自动互通。

### 4.4 奖励可配置

会员等级、折扣、佣金比例、封顶金额、提现手续费、最小提现金额、售后冻结期、内容权限等规则必须后台可配置或通过配置表维护，不能散落硬编码。

## 5. 用户端功能需求

### 5.1 登录与注册

#### 需求描述

用户通过微信快捷登录进入小程序。系统应满足微信平台规则，非必要场景不强制获取用户信息。用户在触发购买、下单、查看受限内容、提现等操作时，引导完成注册。

#### 功能点

1. 游客可进入首页浏览公开内容。
2. 用户注册前必须勾选《用户协议》和《隐私协议》。
3. 注册时必须绑定上级 ID 或邀请码。
4. 系统记录注册来源，包括首页分享、商品分享、邀请码、后台创建。
5. 后台创建的种子账号可无上级。
6. 登录后展示默认头像、用户 ID、实名认证状态、当前等级。
7. 微信头像获取不到时展示默认头像。

#### 验收标准

1. 无推荐关系的普通用户不能完成注册。
2. 后台创建的种子账号可正常登录并生成邀请码。
3. 分享入口进入时，注册后自动绑定分享人。
4. 用户协议和隐私协议未勾选时不能提交注册。

### 5.2 实名认证

#### 需求描述

实名认证用于提现、招商专区购买等关键场景。一期采用人工审核，不对接第三方认证系统。

#### 功能点

1. 用户提交姓名、身份证号、身份证照片。
2. 系统校验身份证号唯一性，已存在则不能提交或审核通过。
3. 后台人工审核认证资料。
4. 实名认证能力支持后台低代码配置开关。
5. 提现前必须完成实名认证。
6. 购买招商专区商品前必须完成实名认证。
7. 认证状态包括未提交、审核中、已通过、已驳回。
8. 驳回时展示驳回原因，用户可重新提交。

#### 验收标准

1. 未实名用户点击提现时弹出实名认证提示。
2. 未实名用户购买招商专区商品时弹出实名认证提示。
3. 同一身份证号不能被多个账号认证通过。
4. 关闭实名认证配置后，受限场景按配置放行或按业务提示处理。

### 5.3 首页

#### 需求描述

首页承载商城入口、专区切换、轮播、公告和客服能力。会议要求重点设计四个专区的切换交互。

#### 功能点

1. 顶部轮播图，支持 3 到 5 张，由后台配置，支持自动轮播和手动滑动。
2. 公告入口，展示平台公告信息。
3. 右侧固定悬浮客服入口，接入小程序客服消息。
4. 首页展示多个专区入口：主产品专区、招商专区、零售专区、礼包专区。
5. 专区切换可采用金刚区、分段控件或标签页，需在 UI 设计阶段确定。
6. 每个专区可配置独立商品、排序和展示状态。

#### 验收标准

1. 后台上下架轮播图后，小程序按配置展示。
2. 公告发布后，小程序公告列表和详情可查看。
3. 用户可从首页进入四个专区。
4. 不同专区展示对应商品，不混用奖励规则。

### 5.4 商品与专区

#### 5.4.1 主产品专区

主产品专区用于主线酒水商品销售、推客升级、主产品推荐奖励和联营商自购折扣。

功能点：

1. 支持商品分类，如系列 A、系列 B。
2. 支持关键词模糊搜索。
3. 支持按价格区间、销量等条件筛选或排序。
4. 支持商品详情页、数量修改、加入购物车、立即购买。
5. 支持购物车结算。
6. 全部包邮，不涉及配送模板。
7. 支持按用户等级展示折扣价。
8. 购买主产品可赠送积分。
9. 用户在主产品区累计消费满 396 元后升级为推客。
10. 主产品区自购或直推购买达到升级门槛时，可升级为县级联营商。

验收标准：

1. 普通会员主产品累计实付满 396 元后自动成为推客。
2. 推客在主产品区复购可看到 8 折价格。
3. 县级、市级、省级联营商可看到对应等级折扣价。
4. 零售专区消费不计入主产品区升级业绩。

#### 5.4.2 招商专区

招商专区用于县级、市级、省级联营商礼包销售和招商奖励计算。

功能点：

1. 招商专区商品为套餐商品。
2. 套餐对应等级：县级联营商、市级联营商、省级联营商。
3. 用户购买招商专区商品前必须实名认证。
4. 支持用户直接购买高等级礼包跳级。
5. 已购买低等级礼包的用户升级时按规则补差价。
6. 招商专区商品可触发招商奖励、扶商奖励、广告奖励。
7. 招商专区购买自身返 5% 广告奖，进入 DF。
8. 招商大额支付可采用线下对公转账加后台手动增加 DF 或人工确认订单的方案。
9. 联营商升级时需要展示并确认合作声明，不对接电子合同。

验收标准：

1. 未实名用户不能购买招商专区商品。
2. 购买县级、市级、省级礼包后，用户等级按套餐升级。
3. 跳级购买后用户直接变更为目标等级。
4. 补差价规则按配置执行。
5. 非招商专区购买不触发招商奖励。

#### 5.4.3 零售专区

零售专区用于普通零售商品销售，只做直推奖励。

功能点：

1. 所有等级看到的价格一致。
2. 只计算直推人奖励，比例为 10%。
3. 奖励直接进入余额。
4. 零售专区订单不累计业绩。
5. 零售专区订单不参与招商奖金制度计算。
6. 零售专区可赠送积分，具体数量由后台商品配置。

验收标准：

1. 零售专区下单后只给直接推荐人发放奖励。
2. 间推或更高层级不获得零售奖励。
3. 零售专区消费不影响推客、联营商升级业绩。

#### 5.4.4 礼包专区

礼包专区为积分兑换区。

功能点：

1. 仅支持纯积分兑换。
2. 不支持积分加现金。
3. 不支持积分加余额、积分加酒豆等组合支付。
4. 积分不足时不能兑换，并提示积分不足。
5. 商品可由后台配置所需积分。

验收标准：

1. 积分足够时可提交兑换订单。
2. 积分不足时不能提交。
3. 礼包专区订单不触发佣金奖励。

### 5.5 内容中心

#### 需求描述

内容中心提供视频和图文内容，可设置观看权限和付费规则。

#### 功能点

1. 视频列表展示标题、缩略图、播放次数。
2. 视频详情展示标题、内容、播放次数、发布时间。
3. 视频支持横屏观看和拖动进度条。
4. 图文列表展示标题、缩略图、浏览量、发布时间。
5. 图文详情展示标题、浏览量、发布时间和正文内容。
6. 后台可按等级设置内容展示权限。
7. 后台可设置内容是否付费。
8. 付费方式支持余额或在线支付。
9. 内容页仅保留点赞、浏览数或播放数，不做评论和回复。

#### 验收标准

1. 不满足等级权限的用户不能查看受限内容。
2. 付费内容支付成功后可查看。
3. 已购买内容再次进入无需重复支付。
4. 评论和回复入口不出现在一期页面。

### 5.6 我的资产

#### 5.6.1 余额

余额主要来源于直推佣金奖励和后台增加，可提现或购买商品。

功能点：

1. 余额分为可提现余额和冻结余额。
2. 订单未过售后期时，佣金进入冻结余额。
3. 售后期结束后，冻结余额转为可提现余额。
4. 余额可用于商品购买。
5. 余额可发起提现。
6. 后台可增加或减少余额。
7. 用户可查看余额变动明细，包含日期、数量、类型、关联订单或备注。

验收标准：

1. 奖励产生后先进入冻结余额。
2. 订单满足解冻条件后转入可提现余额。
3. 提现审核通过后余额扣减。
4. 后台调整余额后生成明细和操作日志。

#### 5.6.2 DF

DF 来源于招商奖励、扶商奖励、广告奖励、后台增加和他人赠送。DF 不可提现。

功能点：

1. DF 可兑换为酒豆。
2. DF 可赠送他人。
3. 后台可增加或减少 DF。
4. 用户可查看 DF 变动明细。
5. DF 不支持提现。

验收标准：

1. 用户不能对 DF 发起提现。
2. DF 兑换酒豆后，DF 扣减，酒豆增加。
3. DF 赠送后，赠送方扣减，接收方按规则增加。

#### 5.6.3 酒豆

酒豆在材料中也被称为“九豆”，一期需先统一产品命名。本文暂称酒豆。

功能点：

1. 酒豆来源于 DF 兑换、后台增加或他人赠送 DF 后形成的可用资产。
2. 酒豆不可提现。
3. 酒豆可用于购买指定专区商品。
4. 酒豆与专区绑定，不同专区是否可用由后台配置。
5. 酒豆不足时提示酒豆不足。
6. 酒豆不与余额、现金或积分组合支付。
7. 用户可查看酒豆变动明细。

验收标准：

1. 支持酒豆支付的商品可选择酒豆支付。
2. 不支持酒豆支付的商品不展示酒豆支付入口。
3. 酒豆不足不能提交订单。
4. 酒豆支付订单不再叠加其他支付方式。

#### 5.6.4 积分

积分主要用于礼包专区兑换。

功能点：

1. 购买商品可赠送积分。
2. 后台可按单个商品配置赠送积分数量。
3. 积分可用于礼包专区兑换商品。
4. 积分可支持未来签到、抽奖等玩法，但一期不开发。
5. 用户可查看积分变动明细。

验收标准：

1. 商品支付完成后按配置发放积分。
2. 取消、退款或售后完成后按规则回滚积分。
3. 积分兑换扣减后生成明细。

#### 5.6.5 期权

报价清单中包含期权资产，但会议未明确其业务用途。一期建议仅保留后台手工加减和明细展示，是否进入用户端需业务确认。

### 5.7 我的团队

#### 功能点

1. 展示我的邀请人。
2. 展示直推人数和直推列表。
3. 展示间推人数和间推列表。
4. 展示团队总人数统计。
5. 列表字段包含用户 ID、消费金额、当前等级、注册时间。
6. 直推列表支持对用户赠送 DF。
7. 小程序端最多展示直推和间推明细。
8. 不展示三级及以上人员列表。

#### 验收标准

1. 用户能看到自己的邀请人。
2. 用户能看到直推、间推列表。
3. 三级及以上人员不出现在小程序列表。
4. 团队总人数可以包含更深层级，但不暴露明细。

### 5.8 订单

#### 功能点

1. 订单状态包括待付款、待发货、已发货、已完成、已取消、售后中。
2. 待付款订单支持付款和取消。
3. 待发货订单支持取消申请，具体是否允许按后台配置或业务规则执行。
4. 已发货订单展示物流信息。
5. 已完成订单支持申请售后。
6. 用户可确认收货。
7. 评价功能在报价清单中出现，但会议未确认，一期建议暂不做或仅保留入口隐藏。
8. 订单支付方式包括在线支付、余额支付、酒豆支付、积分兑换，具体由商品和专区决定。

#### 验收标准

1. 不同订单状态展示正确操作按钮。
2. 后台录入物流单号后，用户端订单详情能看到物流单号和发货状态。
3. 未付款订单取消后不发放奖励和积分。
4. 已支付订单取消或售后完成后，资产和奖励按规则回滚。

### 5.9 售后

#### 功能点

1. 用户可在线提交售后申请。
2. 售后类型支持退货、换货。
3. 换货申请可选择换货商品。
4. 后台审核售后申请。
5. 线下处理退货、换货和打款。
6. 线上售后状态简化为已申请、待发货、已发货、已完成、已关闭。
7. 后台可提前结束售后。

#### 验收标准

1. 用户提交售后后，后台可审核通过或拒绝。
2. 审核结果在用户端可见。
3. 售后完成后订单和资产状态正确更新。
4. 售后期内相关佣金保持冻结。

### 5.10 地址

#### 功能点

1. 地址列表。
2. 地址详情。
3. 新增、删除、修改地址。
4. 设置默认地址。
5. 下单时选择配送地址。

#### 验收标准

1. 默认地址在结算页自动带出。
2. 删除默认地址后，系统按规则重新指定或提示用户选择。
3. 下单时必须有有效收货地址。

### 5.11 设置

## 6. 奖励和等级规则

### 6.1 推客规则

| 项目 | 规则 |
| --- | --- |
| 晋升条件 | 主产品区累计消费满 396 元 |
| 直推首单佣金 | 20%，仅主产品区 |
| 直推复购佣金 | 10%，会议已明确；如后续统一调整，以后台配置为准 |
| 自购权益 | 主产品区 8 折 |
| 招商资格 | 无 |
| 招商区推荐奖励 | 无 |

### 6.2 联营商升级规则

联营商可通过两类路径升级：

1. 主产品区自购或直接推荐购买达到指定金额。
2. 招商区购买对应等级礼包。

会议中出现两组招商礼包金额口径：

1. 报价清单和会议摘要均出现县级 1 万、市级 4 万、省级 20 万。
2. 待办中出现从 5 万升级到 15 万的补差描述。

因此金额应先配置化，最终金额需业务确认后生效。

### 6.3 联营商权益

| 等级 | 主产品区折扣 | 主产品首单佣金 | 主产品复购佣金 | 招商奖励 | 扶商奖励 | 广告奖 |
| --- | --- | --- | --- | --- | --- | --- |
| 县级联营商 | 6.6 折 | 20% | 待确认 | 20%，每单 6000 元封顶 | 招商奖励的 10% | 招商区购买人自身返 5% 到 DF |
| 市级联营商 | 6 折 | 20% | 待确认 | 20%，每单 20000 元封顶 | 招商奖励的 10% | 招商区购买人自身返 5% 到 DF |
| 省级联营商 | 5 折 | 20% | 待确认 | 20%，不封顶 | 直推收入或招商奖励口径待确认 | 招商区购买人自身返 5% 到 DF |

复购佣金比例存在新旧口径：

1. 报价清单旧口径：县级 10%、市级 12%、省级 15%。
2. 5.12 待办新口径：从 10/12/15 调整为 5/8/10。

产品建议：一期系统按等级配置比例，不硬编码。默认配置需由业务在上线前确认。

### 6.4 招商奖励追溯规则

1. 招商奖励仅由招商专区购买触发。
2. 如果购买人上级链路中存在有资格的联营商，则向上追溯到符合条件的联营商发放招商奖励。
3. 中间无等级用户不拿招商奖励。
4. 下级团队用户再次购买招商区商品时，招商奖励仍按规则发放。
5. 通过主产品区自购或主产品区直推业绩升级为联营商时，上级不获得招商奖励。

### 6.5 扶商奖励规则

1. 扶商奖励发给获得招商奖励者的上级联营商。
2. 若上级不是联营商，则不发放扶商奖励。
3. 奖励比例为招商奖励的 10%，但省级口径在材料中存在差异，需确认。
4. 下级团队再次购买招商区商品时，扶商奖励仍按规则发放。

### 6.6 跳级与补差

1. 用户允许直接购买高等级招商礼包跳级。
2. 已通过招商区购买低等级礼包的用户，可购买更高等级礼包补差升级。
3. 已通过主产品区自购或直推业绩升级的用户，如需继续升级，应购买对应招商区升级礼包，具体是否补差需业务确认。
4. 县级升级到省级是否允许一次补差，材料中倾向不允许直接补差到省级，需按业务最终配置执行。

## 7. 支付与提现

### 7.1 支付方式

| 支付方式 | 适用范围 | 规则 |
| --- | --- | --- |
| 在线支付 | 普通商品、内容付费等 | 微信支付，具体接口按现有系统能力 |
| 余额支付 | 支持余额支付的商品 | 不与酒豆或积分组合 |
| 酒豆支付 | 指定专区或指定商品 | 酒豆不足则不能支付 |
| 积分兑换 | 礼包专区 | 纯积分支付 |
| 线下对公转账 | 大额招商或充值 | 线下完成后由后台人工确认或增加 DF |

### 7.2 提现

功能点：

1. 提现来源为可提现余额。
2. 冻结余额不可提现。
3. 提现前必须实名认证。
4. 后台可设置提现手续费。
5. 后台可设置最小提现金额。
6. 初期可线下打款。
7. 后续可开通微信零钱自动打款，减少人工干预。
8. 提现申请需后台审核。

验收标准：

1. 未实名用户不能提现。
2. 低于最小提现金额不能提现。
3. 提现申请审核通过后，可提现余额扣减。
4. 审核拒绝后余额不扣减或退回。
5. 每笔提现有状态和操作记录。

## 9. 数据迁移

### 9.1 背景

旧系统数据需要导入新平台，但旧数据结构与新平台存在差异，需要清洗和手动处理。

### 9.2 一期要求

1. 业务侧暂停旧系统新增录入，避免迁移期间数据继续分叉。
2. 梳理旧系统用户、推荐关系、等级、资产、订单、商品等数据字段。
3. 输出迁移字段映射表。
4. 清洗无上级、重复手机号、重复身份证、异常等级、异常资产等数据。
5. 迁移前生成备份。
6. 迁移后进行抽样核验和总量核对。

### 9.3 验收标准

1. 迁移前后用户总数、资产总额、等级数量有核对表。
2. 异常数据有处理记录。
3. 种子账号和推荐链路迁移后可正常用于注册和奖励计算。

## 10. 合规与风控

### 10.1 推荐关系展示

1. 小程序端最多展示直推和间推。
2. 小程序端不能展示三级及以上人员明细。
3. 小程序端不能展示多层级树状图。
4. 小程序端文案避免强调多层级收益、团队裂变收益、无限收益。

### 10.2 奖励计算

1. 零售专区只给直推发放奖励。
2. 主产品区按直推首单和复购规则发放。
3. 招商奖励与扶商奖励必须与专区隔离。
4. 奖励结算应保留可审计记录。
5. 奖励比例、封顶金额和发放对象需配置化。

### 10.3 实名认证

1. 酒类行业是否强制实名需要补充政策依据或平台提示文案。
2. 在依据未明确前，可将实名认证作为提现、招商购买等高风险操作的业务风控要求。

### 10.4 资金安全

1. 冻结余额用于售后保障。
2. 人工资产调整必须有原因和日志。
3. 提现需审核。
4. 大额线下转账必须有凭证、审核和入账记录。

## 11. 非功能需求

### 11.1 安全

1. 用户端接口必须校验登录态和用户权限。
2. 后台接口必须校验管理员权限。
3. 身份证照片等敏感信息需要限制访问。
4. 资产调整、提现审核、实名审核等关键操作需记录操作日志。
5. 支付回调、订单状态变更、奖励发放需要幂等处理。

### 11.2 可配置性

以下规则必须配置化：

1. 等级名称和等级顺序。
2. 晋升条件。
3. 主产品折扣。
4. 佣金比例。
5. 招商奖励封顶金额。
6. 扶商奖励比例。
7. 广告奖比例。
8. 提现手续费和最小提现金额。
9. 售后冻结期。
10. 内容观看权限。
11. 商品可用支付方式。

### 11.3 可审计性

以下行为必须有日志：

1. 后台创建账号和邀请码。
2. 实名审核。
3. 用户等级变更。
4. 订单状态变更。
5. 奖励生成、冻结、解冻、回滚。
6. 余额、DF、酒豆、积分、期权调整。
7. 提现审核。
8. 系统配置变更。

### 11.4 性能

1. 首页、商品列表、商品详情、订单列表应满足常规小程序体验。
2. 团队统计可能涉及多层推荐关系，应使用异步统计或冗余统计表，避免用户端实时递归查询。
3. 奖励结算应采用事务或可靠任务，避免重复发放。

### 16.2 登录拦截与注册绑定

1. 公开可访问：首页、商品列表、商品详情、公告列表、公告详情、公开内容列表、帮助中心。
2. 必须登录：购物车、结算、订单、售后、地址、资产、提现、团队、邀请海报、实名认证、付费内容购买。
3. 必须完成推荐绑定：普通用户注册、首次进入会员中心、首次下单前。后台创建的种子账号除外。
4. 无推荐参数注册时，展示邀请码绑定页，不允许跳过；邀请码无效、已禁用、已过期、已达使用次数时展示明确错误。
5. 分享小程序、分享商品、邀请海报、邀请码图片进入时，客户端需缓存推荐参数至注册完成，后端以一次性绑定结果为准。
6. 已绑定上级的用户再次通过他人链接进入，不得覆盖原邀请关系。

### 16.3 商品、购物车和结算交互

1. 商品列表支持关键词、专区、分类、价格区间、销量、新品、综合排序；筛选项由后端返回，前端不得假设固定分类。
2. 商品空状态需区分“无搜索结果”“当前专区暂无商品”“网络错误”，并提供清空筛选或重试操作。
3. 商品详情展示商品状态、库存、规格、专区、可用支付方式、等级价、赠送积分、奖励提示。所有比例、金额、等级文案从后端商品详情和配置接口获取。
4. 加入购物车前校验登录、商品上架、库存、规格和购买限制；失效商品在购物车中标灰并不可结算。
5. 结算页展示地址、商品、数量、运费、优惠、资产抵扣、实付金额、预计奖励提示。预计奖励只作预估，最终以后端奖励记录为准。
6. 一笔订单只能选择一种支付方式。余额、DF、酒豆、积分、在线支付不得组合使用；礼包专区仅纯积分兑换。
7. 结算提交后后端锁定价格、库存、支付方式和订单金额；前端刷新页面不得重复创建订单。

### 16.4 支付结果验收

1. 支付方式页展示订单号、倒计时、应付金额、可用支付方式、不可用原因。不可用原因包括余额不足、酒豆不足、积分不足、商品不支持、实名未通过、订单状态不允许支付。
2. 微信支付成功后进入支付成功页，展示订单号、支付金额、支付方式、获得积分或预计奖励提示，并提供“查看订单”“返回首页”。
3. 余额、酒豆、积分支付成功后同样进入支付成功页，并保证资产扣减和订单状态在同一事务内完成。
4. 支付失败页展示失败原因、剩余支付时间、订单号、金额、支付方式，并提供“重新支付”“取消订单”。
5. 支付超时、用户取消、回调验签失败、重复回调、订单已关闭、金额不一致必须进入明确错误态或后台告警，不得静默成功。
6. 前端支付结果只展示后端查询结果；不得仅凭微信客户端回调判断最终支付成功。

### 16.5 物流和售后验收

1. 物流详情展示订单状态、物流公司、物流单号、发货时间、收货地址、商品列表。未对接第三方物流 API 时，仅展示后台录入信息和平台处理节点。
2. 后台未发货时，物流页展示“暂未发货”；物流单号缺失时不展示虚假轨迹。
3. 售后申请支持选择售后商品、数量、售后类型、原因、说明、图片凭证。可申请售后的状态、时限和冻结期由后端配置返回。
4. 售后详情展示申请信息、审核状态、审核备注、拒绝原因、线下处理结果、退款或换货说明、创建时间、更新时间。
5. 售后状态变化必须同步影响订单状态、奖励冻结/回滚、积分回滚和资产流水。

### 16.6 邀请码、邀请海报和 DF 赠送

1. 我的邀请码页展示邀请码、有效状态、可复制操作、分享小程序、生成海报入口。邀请码由后端生成，前端不得本地生成。
2. 邀请海报二维码必须包含推荐人标识、渠道来源和有效期参数；扫码注册后绑定关系以服务端解析和校验为准。
3. DF 赠送页支持通过用户 ID、手机号或邀请码查找接收人；提交前展示接收人脱敏信息、赠送数量、手续费或到账规则。手续费或到账规则如未确认，则接口返回“待业务确认/不可提交”。
4. DF 赠送必须二次确认，校验可用 DF、接收人有效性、幂等请求号；赠送成功后生成赠送方扣减流水、接收方增加流水和赠送记录。
5. DF 赠送记录展示方向、对方用户、数量、状态、时间、失败原因；DF 不可提现。

## 21. 资产独立、奖励审计和幂等要求

1. 余额、冻结余额、DF、酒豆、积分、期权分别独立账户、独立流水、独立变动原因；不得出现余额加酒豆、积分加现金、积分加酒豆等组合支付。
2. 所有资产写操作必须在事务内完成账户余额更新、流水写入、业务单状态更新和操作日志写入。
3. 所有奖励记录必须保存来源订单、来源用户、受益用户、推荐链快照、规则快照、资产类型、冻结状态、幂等键、创建时间、解冻/回滚时间。
4. 支付回调、订单完成、奖励发放、奖励解冻、售后回滚、提现审核、DF 赠送、后台资产调整必须具备幂等处理。
5. 后台人工调整资产必须上传或填写凭证说明，系统记录调整前后值，且不可物理删除流水。
6. 提现审核通过后冻结或扣减可提现余额，线下打款记录需包含打款时间、打款账户、流水号、实际到账金额和操作人。
7. 售后期内奖励保持冻结；售后关闭或超过冻结期后按配置解冻；退款或售后完成后按规则回滚奖励和积分。
8. 深层推荐链只用于奖励计算、统计和审计；小程序接口不得返回三级及以上人员明细。

## 24. 风险点补充

1. 合规风险：设计稿中存在“合伙人等级”“团队分润”等表达，容易与一期合规边界冲突；一期必须隐藏合伙人制度，团队页禁止三级以上明细和树状关系。
2. 配置风险：若比例、金额、封顶、补差规则未确认而种子数据写入默认值，可能造成上线后错误结算；待确认配置不得自动生效。
3. 资金风险：余额、DF、酒豆、积分混用或组合支付会破坏资产独立原则，必须由订单预览和支付接口强约束。
4. 审计风险：奖励和资产若不保存规则快照，配置变更后历史订单难以复盘。
5. 幂等风险：支付回调、DF 赠送、提现审核、奖励发放和售后回滚重复执行会导致重复扣减或重复发放。
6. 体验风险：样式稿页面较多，若缺通用状态组件，空列表、网络错误、未登录拦截会出现体验不一致。
7. 范围风险：体验店、优惠券、消息中心、系统更新等设计页容易扩大一期范围，应在需求冻结时明确是否隐藏。
